## 2. Product Overview

Build a **100% local web application** using **vanilla JavaScript, HTML, and CSS** that allows a user to:

1. Record **system audio** from the computer  
2. Record audio from one or more **attached microphones**  
3. Mix all selected audio inputs into a **single output file**

5. Display a **live equalizer-style audio visualization while recording**  
6. Enable downloading the final file only after the user presses **Stop** and recording has been finalized  

The application must run entirely **in the browser with no backend, no cloud services, and no external processing**. All audio capture, mixing, visualization, encoding, and file generation must happen locally on the user’s machine.

The UI should be simple and task-focused, with an expandable **Options** panel where users can configure relevant recording settings. User preferences should persist across sessions using a **cookie**.

## 12. Suggested Implementation Notes (For Claude Code)

Use this guidance when implementing:

### Audio Capture
- Use `getUserMedia()` for microphone input  
- Use `getDisplayMedia()` for system audio capture when enabled and available  
- Route both into a `Web Audio API` graph  
- Mix sources through gain nodes  
- Downmix to mono before sending to recorder output  

### Recording
- Create a mixed destination stream using `MediaStreamAudioDestinationNode`  
- Feed the mixed stream into `MediaRecorder`  
- Collect chunks and assemble a Blob on stop  
- Create a downloadable object URL for the final file  

### Compression / Speech Optimization
- Prefer mono output  
- Use lower `audioBitsPerSecond` or equivalent supported settings  
- Prioritize browser-supported compressed formats  
- Favor intelligibility and storage efficiency over fidelity  

### Equalizer Visualization
- Add an `AnalyserNode` to the mixed signal chain  
- Use `getByteFrequencyData()` (preferred for equalizer-style bars) or `getByteTimeDomainData()` if needed  
- Render to a `<canvas>` element during recording  
- Drive redraws with `requestAnimationFrame`  
- Stop animation loop cleanly on stop or error  
- Keep visualization decoupled from recording logic so visualization failure does not block recording  

### State Management
Use a simple front-end state machine:
- idle  
- recording  
- processing  
- ready  
- error  

### Options Persistence
- Store settings as JSON in a cookie  
- Load cookie on startup  
- Validate restored values before applying  

### UI Behavior
- Download button disabled until Blob/object URL exists  
- Options panel should be toggleable  
- Show current recording status clearly  
- Show permission and compatibility errors inline  
- Visualization area should be visible in the main interface during recording
